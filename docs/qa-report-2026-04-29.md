## Issues Found and Fixed

### ISSUE-003 (CRITICAL — Fixed) — New resume created on every save
- **Symptom:** Each "Next" click or auto-save created a NEW resume in the DB; 6 orphaned resumes accumulated in one session.
- **Root cause:** `ResumeContext.updateResume` spread `data` without including `id` into the optimistic state object (`const optimistic = { ...data, ... }`). After the first save, `currentResume.id` became `undefined`. The next save saw no ID → called `createResume` instead of `updateResume`.
- **Fix:** `src/context/ResumeContext.jsx:51` — `const optimistic = { id, ...data, updatedAt: ... }`
- **Commit:** 7235701
- **Verified:** DB shows exactly 1 resume for the QA test user after completing all 7 steps.

### ISSUE-006 (Fixed) — Preview page shows "Resume not found" while loading
- **Symptom:** Navigating directly to `/app/preview/:id` showed "Resume not found" immediately (before Supabase returned data).
- **Root cause:** `resumes` array starts as `[]`; component checked `!resume` before async load completed.
- **Fix:** `src/pages/preview/ResumePreviewPage.jsx` — added `loadingResumes` guard before the not-found check.
- **Commit:** 1c5d245

### ISSUE-004 (Fixed) — User menu dropdown stays open on outside click
- **Symptom:** The Settings/Sign Out dropdown in the Navbar had no click-outside handler, so it stayed visible when interacting elsewhere on the page.
- **Fix:** `src/components/layout/Navbar.jsx` — added `useRef` + `useEffect` with `mousedown` listener.
- **Commit:** dc1329b

### ISSUE-007 (Fixed) — Step indicator only allows clicking current/past steps
- **Symptom:** On a fresh builder load (step 0), all steps except step 1 were non-clickable. There was no way to jump to a previously visited step.
- **Fix:** `src/components/common/StepIndicator.jsx` — added `maxStep` prop; steps are clickable if `i <= maxStep`. `ResumeBuilderPage.jsx` tracks `maxStep` and advances it on each Next click.
- **Commit:** dc1329b

### ISSUE-007b (Fixed) — Step advancement regression introduced by ISSUE-007 fix
- **Symptom:** After the ISSUE-007 fix, clicking "Next" didn't advance to the next step because the `useEffect` with `[id, resumes, setCurrentResume]` dependencies was re-running on every auto-save (which updates `resumes`), resetting `step` back to `current_step` from the DB.
- **Fix:** `src/pages/resume-builder/ResumeBuilderPage.jsx` — added `stepRestoredRef` to gate the step-restore logic to only run once per mount.
- **Commit:** 9c46c40

## Issues Remaining (Open)

### ISSUE-002 (Open) — Duplicate React key warnings (×2) on builder load
- **Symptom:** Browser console shows two "Encountered two children with the same key, `%s`" warnings when the builder page first loads at `/app/builder`.
- **Source:** Not definitively identified from code inspection. All statically-defined key sources (`STEP_LABELS`, `COUNTRIES`, `INDUSTRIES`, `PROFICIENCY_LEVELS`, `MONTHS`) have unique values. Likely from a dynamic data source (e.g., ATS score returning duplicate keywords in `missingKeywords` array, or a runtime data shape mismatch).
- **Impact:** Low — cosmetic warning, no functional breakage.
- **Action needed:** Add browser devtools session with React DevTools to see the exact component stack and key value.

### ISSUE-001 (Open) — Email validation rejects non-standard TLDs
- **Symptom:** Sign-up with `qa.test.scopus@mailtest.dev` was rejected with "Email address is invalid".
- **Root cause:** Supabase auth project configuration rejects certain TLDs. Not a frontend code issue.
- **Impact:** Low — only affects test accounts with unusual TLDs. Standard Gmail/Outlook addresses work fine.
- **Action needed:** Configure Supabase project to allow broader TLD range, or update client-side validation to not pre-reject addresses before Supabase validates them.
